# CLAUDE.md - MangoView

## 项目概述

MangoView 是一个投资决策辅助工具，定位"一笔好决策，只需要看三件事：时机对不对、公司行不行、自己有没有犯傻"。品牌：Mangfolio。

## 技术栈

- 前端：Next.js 14 + Tailwind CSS（Stripe 风格）
- 后端：Python FastAPI
- 数据库：SQLite
- 数据源：akshare（宏观/北向资金/行业板块）

## Superpowers 管理机制

### 核心原则（Process over Prompt）

**固定流程 > 自由发挥**。每次任务必须按流程走，不跳过。

---

### 七阶段流程

```
brainstorming → writing-plans → subagent-dev → tdd → code-review → debug → finishing
```

#### 1. brainstorming（头脑风暴）

**触发**：用户说"做XX"或"帮我改XX"

**必须做**：
- 搞清楚需求：做什么/给谁用/在哪用/要什么输出/限制是什么
- 不明确需求不开始
- 识别关键决策点，列出选项让用户确认

**硬性约束**：需求不明，不动手

#### 2. writing-plans（写计划）

**必须做**：
- 把大任务拆成 2-5 分钟的小块
- 每个小块有明确输入输出
- 验证标准要清晰

**为什么**：大任务会让 AI 焦虑，容易漏步骤

#### 3. subagent-dev（子代理开发）

**触发**：需要并行执行多个任务

**必须做**：
- 设计方案必须先让用户审批
- 用户批准设计方案之前，**绝对不能**写代码
- 两阶段审查：同行审查 → 专家审查

**硬性约束**：设计方案未审批，不写代码

#### 4. tdd（测试驱动）

**必须做**：
- RED：先写一个会失败的测试
- GREEN：写最少的代码让测试通过
- REFACTOR：重构，然后继续下一轮

**硬性约束**：不先写测试，不写代码

#### 5. code-review（代码审查）

**必须做**：
- 每次提交都审查，不是最后才审
- 审查通过才能继续下一任务

**审查内容**：逻辑/风格/安全性/可维护性

#### 6. debug（系统化调试）

**必须按流程**：
1. 复现问题
2. 定位原因
3. 验证假设
4. 修复
5. 确认修复有效

**硬性约束**：不靠直觉，按步骤来

#### 7. finishing（完成分支）

**必须做**：
- 所有测试通过
- 代码审查通过
- 合并分支
- 清理临时文件
- 更新 PROGRESS.md

---

### 四大原则

| 原则 | 说明 |
|------|------|
| **TDD 优先** | 先写测试再写代码，不测试不写码是铁律 |
| **系统化调试** | 不靠猜，按流程排错是工程化调试 |
| **简化复杂性** | 能简单就不复杂，简单是最强的设计原则 |
| **证据优于声明** | 做完要验证，不能嘴上说成功，要有证据 |

---

### 硬性规定（不是建议）

| 规则 | 错误写法 | 正确写法 |
|------|----------|----------|
| 设计方案审批 | "建议先设计方案" | "用户批准设计方案之前，**绝对不能**写任何代码" |
| TDD 流程 | "可以先写测试" | "**必须**先写 RED 测试，再写代码通过 GREEN" |
| 代码审查 | "建议审查代码" | "审查**通过才能**继续下一任务" |
